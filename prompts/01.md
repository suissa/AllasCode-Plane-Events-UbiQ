Você é um Agent de implementação sênior trabalhando no projeto QUICMQ/UbiQ.

Sua tarefa é transformar o QUICMQ em um package instalável e auto-descoberto pelo UbiQ, funcionando como módulo dinâmico de mensageria, sem exigir que o projeto consumidor faça import direto do QUICMQ.

O objetivo final é que um projeto consiga instalar:

```bash
npm install @allascode/quicmq
```

e o UbiQ consiga descobrir automaticamente esse módulo, carregar seu manifest, iniciar o runtime/sidecar do QUICMQ e usar os protocolos configurados sem que o código da aplicação precise importar `@allascode/quicmq`.

A aplicação deve continuar usando apenas a API semântica do UbiQ, por exemplo:

```ts
UbiQ.publish("order.created", payload)

UbiQ.subscribe("order.created", async (event) => {
  // processa evento
})
```

O QUICMQ deve ser tratado como um runtime interno de transporte, não como SDK de aplicação.

---

## 1. Conceito principal

Implemente o QUICMQ como um módulo instalável do UbiQ com três responsabilidades principais:

1. Expor um broker/sidecar chamado `quicmqd`.
2. Expor uma CLI operacional chamada `quicmq`.
3. Expor um manifest `ubiq.module.yml` para auto-discovery pelo UbiQ.

O projeto consumidor não deve precisar importar classes, funções ou clients específicos do QUICMQ.

Errado:

```ts
import { QuicMqClient } from "@allascode/quicmq"

const client = new QuicMqClient()
client.publish("order.created", payload)
```

Correto:

```ts
UbiQ.publish("order.created", payload)
```

O UbiQ deve resolver internamente que o engine de mensageria configurado é `quicmq`.

## 4. Manifest `ubiq.module.yml`

Crie o arquivo `ubiq.module.yml`.

Ele deve declarar que o QUICMQ é um runtime de transporte do UbiQ.

Exemplo:

```yaml
id: allascode.quicmq
name: QUICMQ
kind: ubiq.transport.runtime
runtime: sidecar

binary:
  command: quicmqd
  healthcheck: quicmq health
  shutdown: quicmq shutdown

provides:
  transports:
    - quic
    - nats
    - grpc
    - websocket

requires:
  security:
    mtls: required
    zero_trust: required
    linear_consume_once: required
    dpop: optional

config:
  default_file: config/default.ubiq.quicmq.yml
  project_override_files:
    - ubiq.yml
    - ubiq.quicmq.yml
    - config/ubiq.yml
    - config/quicmq.yml

autoload:
  enabled: true
  when:
    ubiq_config_has: messaging
```

O UbiQ deve conseguir encontrar esse arquivo automaticamente em:

```txt
node_modules/**/ubiq.module.yml
```

ou pelo campo `ubiq.module` do `package.json`.

## 5. Configuração padrão do UbiQ para QUICMQ

Crie `config/default.ubiq.quicmq.yml`.

O padrão obrigatório deve ser:

```yaml
ubiq:
  messaging:
    engine: quicmq

    protocols:
      - quic
      - nats

    security:
      mtls: true
      mtls_disable_allowed: false

      dpop: true
      dpop_disable_allowed: true

      zero_trust: true
      zero_trust_disable_allowed: false

      linear_consume_once: true
      linear_consume_once_disable_allowed: false

    delivery:
      linear_consume_all: false
      linear_consume_all_zero_trust: false

      ack_timeout_ms: 60000
      dlq_after_ms: 600000

      timeout_autodestroy_required: true
      consumed_confirmation_required: true

    dlq:
      enabled: true
      emit_event: true
      event_suffix: dlq
      supervisor_only: true

    supervisor:
      name: AgentsQueueMessagesSupervisor
      privileged: true
      can_restart_agents: true
      can_create_agents: true
      can_recover_from_local_event_store: true
      can_replay_dlq: true
```

A configuração também deve aceitar:

```yaml
ubiq:
  messaging:
    protocols:
      - quic
      - nats
      - grpc
      - websocket
```

Regra:

Se uma subscription não definir protocolos próprios, ela deve ouvir a mesma fila em todos os protocolos globais configurados.

Exemplo:

```yaml
ubiq:
  messaging:
    protocols:
      - quic
      - nats
      - grpc
      - websocket

    subscriptions:
      - queue: billing
        event: payment.created
        protocols:
          - quic

      - queue: analytics
        event: "#"
```

Resultado esperado:

```txt
billing recebe apenas por QUIC.
analytics recebe por QUIC, NATS, gRPC e WebSocket.
```

Implemente a resolução assim:

```txt
effective_subscription_protocols =
  subscription.protocols
  ?? queue.protocols
  ?? broker.protocols
```

## 6. Invariantes de segurança

Implemente validação forte de configuração.

Estas regras são obrigatórias:

```txt
mTLS nunca pode ser desligado.
ZeroTrust nunca pode ser desligado.
LinearConsumeOnce nunca pode ser desligado.
DPoP pode ser desligado por configuração.
```

Configuração válida:

```yaml
security:
  mtls: true
  dpop: false
  zero_trust: true
  linear_consume_once: true
```

Configuração inválida:

```yaml
security:
  mtls: false
```

Erro esperado:

```txt
Invalid QUICMQ configuration: mTLS is mandatory and cannot be disabled.
```

Configuração inválida:

```yaml
security:
  zero_trust: false
```

Erro esperado:

```txt
Invalid QUICMQ configuration: ZeroTrust is mandatory and cannot be disabled.
```

Configuração inválida:

```yaml
security:
  linear_consume_once: false
```

Erro esperado:

```txt
Invalid QUICMQ configuration: LinearConsumeOnce is mandatory and cannot be disabled.
```

DPoP pode ser:

```yaml
security:
  dpop: false
```

Nesse caso, o broker não deve exigir DPoP por evento, mas deve manter mTLS, ZeroTrust, ACK obrigatório, LinearConsumeOnce, LinearAutoDestroy e proteção contra replay.

## 7. Modos de entrega

Implemente três semânticas de entrega.

### 7.1 LinearConsumeOnce

É o modo mínimo obrigatório.

Cada delivery individual só pode ser consumido uma vez por um subscriber autorizado.

Semântica:

```txt
1. Broker cria delivery_id.
2. Broker escolhe subscriber elegível.
3. Broker envia payload.
4. Subscriber envia ACK.
5. Broker libera chave ou confirmação de processamento.
6. Subscriber envia consumed.
7. Broker marca delivery como finalizado.
8. O mesmo delivery_id nunca pode ser consumido novamente.
```

### 7.2 LinearConsumeAll

Quando configurado:

```yaml
delivery:
  linear_consume_all: true
```

O evento global só é finalizado quando todos os subscribers obrigatórios tiverem consumido o evento.

Semântica:

```txt
1. Publisher emite evento.
2. Broker resolve todos os subscribers obrigatórios.
3. Broker cria um delivery individual por subscriber.
4. Cada subscriber precisa ACKar e consumir.
5. O evento global só finaliza quando todos os deliveries individuais finalizarem.
```

### 7.3 LinearConsumeAllZeroTrust

Quando configurado:

```yaml
delivery:
  linear_consume_all_zero_trust: true
```

Esse modo implica:

```txt
linear_consume_all = true
zero_trust = true
mtls = true
linear_consume_once = true
```

Semântica:

```txt
1. Publisher emite {event}.
2. Broker resolve todos os subscribers elegíveis.
3. Para cada subscriber:
   - cria delivery_id individual;
   - cria delivery_epoch individual;
   - cria payload criptografado individual;
   - cria chave de descriptografia individual;
   - registra PendingDelivery.
4. Broker envia payload_ciphertext para o subscriber.
5. Subscriber envia ACK.
6. Broker valida ACK via mTLS e, se ativo, DPoP.
7. Broker envia a chave de descriptografia somente para aquele subscriber.
8. Subscriber descriptografa, processa e envia {event}.consumed.
9. Broker valida o consumed e a chave usada.
10. Broker destrói a chave.
11. Quando todos os subscribers finalizarem, o evento global é finalizado.
```

## 9. Timeout AutoDestroy

O timeout padrão deve ser 1 minuto.

Configuração:

```yaml
delivery:
  ack_timeout_ms: 60000
```

Semântica:

```txt
1. Broker envia payload_ciphertext para o subscriber.
2. Se o subscriber não enviar ACK dentro de 60 segundos:
   - broker envia {event}.timeout.autodestroy para aquele subscriber;
   - o evento contém a chave de descriptografia que deve ser destruída;
   - o subscriber precisa responder {event}.timeout.autodestroyed.
3. Se o subscriber responder timeout.autodestroyed:
   - broker pode reenviar um novo payload imediatamente.
4. Se o subscriber não responder timeout.autodestroyed:
   - broker só deve reenviar depois do próximo intervalo de 1 minuto.
```

Payload do evento de timeout:

```json
{
  "event_id": "evt_123",
  "delivery_id": "del_456",
  "subscriber_id": "sub_789",
  "decypher_key": "key_to_destroy",
  "reason": "ack_timeout"
}
```

Resposta esperada do subscriber:

```json
{
  "event_id": "evt_123",
  "delivery_id": "del_456",
  "subscriber_id": "sub_789",
  "decypher_key": "key_to_destroy",
  "destroyed": true
}
```

O broker deve validar que a chave retornada corresponde ao delivery pendente.

## 11. AgentsQueueMessagesSupervisor

Implemente o contrato do supervisor.

O supervisor é um subscriber privilegiado.

Configuração:

```yaml
supervisor:
  name: AgentsQueueMessagesSupervisor
  privileged: true
  can_restart_agents: true
  can_create_agents: true
  can_recover_from_local_event_store: true
  can_replay_dlq: true
```

Capabilities:

```txt
ubiq.agent.restart
ubiq.agent.spawn
ubiq.agent.recover_from_local_event_store
ubiq.dlq.replay
```

Semântica:

```txt
1. Supervisor escuta *.dlq.
2. Ao receber DLQ, agrupa mensagens por agent_hash e entity_id.
3. Verifica se existe Agent saudável para aquele agent_hash.
4. Se existir, reenvia as mensagens pendentes quando o Agent assinar a fila.
5. Se não existir, reinicia ou cria um Agent novo.
6. O Agent deve recuperar seu último estado usando seu LocalEventStore.
7. Quando o Agent subir e fizer subscribe nas filas necessárias, o UbiQ deve reenviar as mensagens pendentes.
```

Importante:

```txt
Somente um Agent pode ser client direto da mensageria.
Aplicações comuns não devem ser clients diretos do QUICMQ.
```

## 12. Identidade dos Agents

Cada Agent deve possuir um hash identificador derivado de:

```txt
name + schema + graflow
```

Use canonicalização antes do hash.

Exemplo conceitual:

```txt
agent_schema_hash = sha256(canonical_schema)
agent_graflow_hash = sha256(canonical_graflow)
agent_hash = sha256(agent_name + ":" + agent_schema_hash + ":" + agent_graflow_hash)
agent_instance_id = sha256(server_id + ":" + agent_hash + ":" + counter)
```

Regra de unicidade:

```txt
Somente 1 Agent por server_id + agent_hash + counter pode existir.
```

Exemplo:

```yaml
agent:
  name: BillingAgent
  schema: ./agents/billing/schema.json
  graflow: ./agents/billing/BillingAgent.graflow
  counter: 1
```

Pode haver múltiplos Agents semanticamente idênticos no mesmo servidor, desde que possuam counters diferentes:

```txt
BillingAgent#1
BillingAgent#2
BillingAgent#3
```

Todos compartilham o mesmo `agent_hash`, mas possuem `agent_instance_id` diferente.

## 13. Distribuição por entity_id

Quando existirem múltiplos Agents idênticos vivos, o QUICMQ deve agrupar os eventos da mesma `entity_id` e entregar para apenas um Agent equivalente por vez.

Não use random puro.

Use uma estratégia sticky por entidade, preferencialmente Rendezvous Hash ou Jump Consistent Hash.

Semântica:

```txt
1. Filtrar Agents vivos com o mesmo agent_hash.
2. Usar entity_id como chave de roteamento.
3. Escolher um Agent por Rendezvous Hash.
4. Todos os eventos da mesma entity_id devem ir para o mesmo Agent enquanto ele estiver saudável.
5. Se o Agent cair, redistribuir para outro Agent equivalente.
```

Configuração:

```yaml
agent_delivery:
  identical_agent_grouping: true
  group_by: agent_hash

  entity_sticky_delivery: true
  entity_key: entity_id

  balancing:
    strategy: rendezvous_hash
    fallback: random_live_agent

  uniqueness:
    one_agent_per_server_hash_counter: true
```

## 15. Estado interno de delivery

Modele os estados de delivery como enum ou equivalente:

```ts
type DeliveryState =
  | "created"
  | "payload_sent"
  | "ack_received"
  | "key_released"
  | "consumed"
  | "timeout_autodestroy_sent"
  | "timeout_autodestroyed"
  | "dlq"
  | "replayed"
  | "finalized"
```

Transições válidas:

```txt
created -> payload_sent
payload_sent -> ack_received
payload_sent -> timeout_autodestroy_sent
timeout_autodestroy_sent -> timeout_autodestroyed
timeout_autodestroyed -> payload_sent
ack_received -> key_released
key_released -> consumed
consumed -> finalized
timeout_autodestroy_sent -> dlq
payload_sent -> dlq
dlq -> replayed
replayed -> payload_sent
```

Transições inválidas devem ser rejeitadas.

Exemplo:

```txt
consumed -> payload_sent
```

deve falhar, pois delivery já consumido não pode ser reenviado.

## 24. Importante

Não implemente atalhos que enfraqueçam a semântica.

Não aceite:

```txt
mtls: false
zero_trust: false
linear_consume_once: false
```

Não permita que aplicações comuns assinem DLQ.

Não permita que dois Agents com o mesmo `server_id + agent_hash + counter` subam simultaneamente.

Não permita que delivery já consumido seja reprocessado.

Não permita que chave já destruída seja reutilizada.

Não faça o projeto consumidor importar QUICMQ diretamente.

O QUICMQ deve ser um módulo dinâmico de runtime do UbiQ.
